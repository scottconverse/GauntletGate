A walkthrough audit that won't call a product "clean" when it's broken for new users.

FirstRunWalkthrough drives your app with Playwright and reads your repo as the source of truth — then, unlike an ordinary smoke test, it constructs and verifies the real first-run state and walks your product with its dependencies absent. The new-user dead-ends a dev box hides can't slip past it.

Get it on GitHub Read the manual

The problem

A walkthrough run on the developer's machine audits a product that already works — database seeded, model server running, license activated, settings filled in. So it reports clean. Then a real new user installs it, opens it, and hits a wall on the first screen: a disabled button, a "go install X yourself" message, a blank page.

This skill exists because that happened — a product was reported "near-clean (1 Minor)" while a brand-new user with no model server hit an immediate dead-end. The audit ran on a provisioned dev box, and its "clean profile" isolation had silently failed, so it secretly read the real, working profile.

What it does differently

Verifies the first-run state

Fresh profile, empty data, dependencies absent — and it proves the app actually used the clean state instead of trusting an override that may have silently failed.

Walks with dependencies ABSENT

The new-user reality, where dead-ends live — not just the convenient state where everything is already installed.

A dev box is disqualifying

If a clean first-run state can't be verified, the run is marked INVALID for first-run — it is never allowed to report "clean."

Dead-end = Blocker

A core feature a new user can't reach by following the in-product path is a Blocker, not a footnote.

All of that sits on top of a thorough interface-wiring audit: routes, forms, state, backend wiring, docs/design cross-check, and test reality.

The audit pipeline

Two gates. The amber stages are the first-run discipline ordinary walkthroughs skip; the red branch is where a run disqualifies itself from reporting "clean."

Install & run

python install.py            # → ~/.claude/skills/walkthrough/  (Claude Code, every project)
# or, project-only:
python install.py --project /path/to/your/project

# then, in a fresh session:
/walkthrough                 # or: "do a full UI walkthrough + interface-wiring audit"

You get an evidence-backed report: a verdict that states up front whether a new user can reach the core feature, a verified environment attestation, a zero-state/first-run section and provisioning matrix, numbered findings, a docs/design cross-check, and a test-reality assessment.

Honest about what it is

It's a skill — Markdown instructions an AI agent follows — not a binary or a CI job. It's written for and verified against Claude Code and Codex; the method is agent-agnostic but assumes an agent that can run Playwright and read your repo. It makes the first-run pass mandatory and gates the verdict on a verified attestation — the strongest a skill can be — but it ultimately relies on the agent executing it faithfully. It audits; it doesn't change your code unless you explicitly switch it to repair mode.
